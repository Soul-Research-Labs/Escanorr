# Groth16 Setup Guide

This guide walks through replacing the placeholder Groth16 verification key in `Groth16Verifier.sol` with a production key derived from the recursive proof-wrapping circuit.

> **Prerequisites**: Rust nightly toolchain, `snark-verifier` crate, ~16 GB RAM for SRS operations.

---

## Overview

ESCANORR uses a two-stage proof pipeline for EVM chains:

1. **Halo2/IPA proof** (Pallas/Vesta) — generated by `escanorr-prover`
2. **Groth16/BN254 wrapper** — proves "I verified a valid Halo2 proof" inside a BN254 circuit

The Groth16 step requires a **structured reference string (SRS)** from a trusted setup ceremony. We use the [Perpetual Powers of Tau](https://github.com/privacy-scaling-explorations/perpetualpowersoftau) ceremony maintained by the Ethereum community.

---

## Step 1: Download the SRS

Download the Powers of Tau file matching your circuit size. For the recursive wrapping circuit (estimated k ≈ 20–22):

```bash
# Download the challenge file (k=22 supports up to 2^22 constraints)
curl -L -o powersOfTau28_hez_final_22.ptau \
  https://storage.googleapis.com/zkevm/ptau/powersOfTau28_hez_final_22.ptau

# Verify the hash (check against the ceremony transcript)
sha256sum powersOfTau28_hez_final_22.ptau
```

The SHA-256 hash should match the value published in the [Hermez ceremony transcript](https://github.com/iden3/snarkjs#7-prepare-phase-2).

---

## Step 2: Build the Wrapping Circuit

The wrapping circuit takes a Halo2/IPA proof as a witness and produces a Groth16/BN254 proof. This uses `snark-verifier` from Axiom:

```rust
use snark_verifier::loader::native::NativeLoader;
use snark_verifier::system::halo2::transcript::evm::EvmTranscript;

// 1. Load the Halo2 verification key and proof
let halo2_vk = /* your Halo2 VerifyingKey */;
let halo2_proof = /* serialized proof bytes */;
let halo2_instances = /* public inputs */;

// 2. Create the aggregation circuit
let aggregation_circuit = AggregationCircuit::new(
    &params_bn254,       // BN254 SRS
    vec![halo2_snark],   // The Halo2 proof to verify
);

// 3. Generate Groth16 proving/verification keys
let (groth16_pk, groth16_vk) = keygen(&params_bn254, &aggregation_circuit);
```

> **Note**: The actual snark-verifier integration is tracked in [ADR-003](../adr/003-recursive-proof-wrapping.md). Until `EvmAdapter::submit()` is implemented, this guide describes the target workflow.

---

## Step 3: Export the Verification Key

Extract the Groth16 verification key components for the Solidity contract:

```rust
// Export VK components as hex strings
let alpha = format_g1_point(&groth16_vk.alpha_g1);
let beta  = format_g2_point(&groth16_vk.beta_g2);
let gamma = format_g2_point(&groth16_vk.gamma_g2);
let delta = format_g2_point(&groth16_vk.delta_g2);
let ic    = groth16_vk.ic.iter().map(format_g1_point).collect::<Vec<_>>();
```

The output format must match the `Groth16Verifier.sol` contract's constructor expectations:

- `alpha`: G1 point (2 × uint256)
- `beta`: G2 point (2 × 2 × uint256)
- `gamma`: G2 point
- `delta`: G2 point
- `IC`: array of G1 points (length = number of public inputs + 1)

---

## Step 4: Update the Solidity Contract

Replace the placeholder verification key in `contracts/src/Groth16Verifier.sol`:

```solidity
// Replace these with the values from Step 3
Pairing.G1Point memory alpha = Pairing.G1Point(
    0x..., // alpha.x
    0x...  // alpha.y
);

Pairing.G2Point memory beta = Pairing.G2Point(
    [0x..., 0x...], // beta.x (Fp2)
    [0x..., 0x...]  // beta.y (Fp2)
);

// ... similarly for gamma, delta, and IC points
```

## Step 5: Deploy and Verify

```bash
# Run the Solidity tests to ensure the new VK doesn't break existing tests
cd contracts
forge test -vvv

# Deploy the updated verifier to a testnet
forge script script/DeployVerifier.s.sol --rpc-url $RPC_URL --broadcast
```

## Security Considerations

- **SRS provenance**: Only use SRS files from a verified ceremony with many independent participants. The Hermez/Perpetual Powers of Tau ceremony has >80 contributors.
- **Circuit fingerprinting**: The Groth16 VK is circuit-specific. Any change to the wrapping circuit (e.g., supporting a new Halo2 circuit) requires regenerating the VK and redeploying the verifier contract.
- **No secret material**: The SRS download contains only public parameters. The toxic waste from the ceremony is destroyed by the multi-party protocol.
